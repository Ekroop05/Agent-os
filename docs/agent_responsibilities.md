# Agent Responsibilities

Agent OS is designed as a multi-agent system where different components (agents) take on specific roles. Here is the breakdown of their responsibilities, inputs, outputs, and limitations.

---

## 1. Architect

**Responsibilities:**
- Act as the user-facing technical lead or CTO.
- Gather and extract project requirements through natural conversation.
- Identify missing information and ask clarifying questions.
- Design the high-level system architecture, tech stack, and major components.
- Generate the initial breakdown of planning tasks.

**Inputs:**
- User chat messages.
- The current `ProjectStateManager` context (accumulated requirements).

**Outputs:**
- Conversational replies to the user.
- A structured JSON `architecture` object (tech stack, components, tasks).
- A detailed project specification (via `spec_engine`).

**Dependencies:**
- `requirement_extractor` for parsing keywords.
- LLM (`qwen3:14b`) for conversational responses and architecture design.
- `ProjectStateManager` for maintaining context.

**Limitations:**
- Requires the user to answer questions sequentially.
- Pre-programmed to expect specific fields (frontend, backend, database).
- Cannot easily pivot once the architecture is generated unless a new conversation is started.

---

## 2. Planner (Integrated Orchestrator Logic)

*Note: While there is a `planner_agent.py`, the active planning logic heavily resides within the Architect's approval flow and the `build_orchestrator`.*

**Responsibilities:**
- Convert the high-level architecture into an actionable, sequential list of tasks.
- Enrich task descriptions with context from the project specification.
- Assign tasks to the appropriate execution agents (Builder, Security).
- Maintain the execution order and handle retries if tasks fail.

**Inputs:**
- The approved `architecture` JSON.
- The project `spec`.

**Outputs:**
- Executable `Task` records injected into the `task_service`.

**Dependencies:**
- `task_service` for storing and tracking tasks.
- `build_orchestrator` for managing the execution loop.

**Limitations:**
- Currently creates a somewhat static, linear list of tasks based on the Architect's initial JSON.
- Limited ability to dynamically re-plan or pivot the entire architecture mid-build based on Builder feedback.

---

## 3. Builder

**Responsibilities:**
- Execute the assigned implementation tasks.
- Generate actual folder structures and code files.
- Scaffold boilerplate, build UI components, and write backend logic.
- Ensure the output aligns with the project specification (theme, colors, tone).

**Inputs:**
- Task title and description.
- Project architecture and specification.
- A representation of the existing workspace folder structure.

**Outputs:**
- Raw source code and files written to disk.
- A JSON payload listing the created file paths and their contents.

**Dependencies:**
- LLM (`qwen2.5-coder:7b`) for code generation.
- `path_security` for ensuring writes remain within the workspace.
- Predefined templates (as a fallback if the LLM fails).

**Limitations:**
- Code quality is entirely dependent on the LLM's capabilities.
- Context window limits: It may struggle to maintain awareness of large, complex codebases as the file count grows.
- Often attempts to write the entire feature at once, which can lead to truncated files if the output is too long.

---

## 4. Security

**Responsibilities:**
- Act as the automated QA and Security Reviewer.
- Validate that the Builder successfully completed the task without introducing vulnerabilities.
- Block dangerous code patterns and ensure file system isolation.

**Inputs:**
- The list of files generated by the Builder in the recently completed task.
- The source code of those generated files.

**Outputs:**
- A `SecurityReviewResult` containing an approval status (True/False) and a list of identified issues.

**Dependencies:**
- Regular expressions for identifying dangerous patterns.
- `command_security` and `path_security` modules.
- LLM (`qwen2.5-coder:7b`) for semantic code review (missing imports, broken logic).

**Limitations:**
- Primarily relies on static regex patterns, which can result in false positives or missed obfuscated vulnerabilities.
- The LLM code review is basic and might miss complex logical bugs or deep integration issues across multiple files.
- Rejection feedback to the Builder is sometimes too brief to guide the LLM to a successful retry.
