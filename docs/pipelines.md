# Pipeline Analysis — Agent OS v2

This document provides an in-depth analysis of the five core execution pipelines powering Agent OS. Each pipeline governs a critical phase of autonomous software creation and execution.

---

## 1. Project Creation Pipeline

The Project Creation Pipeline is the overarching sequence that transforms an abstract user idea into a fully scaffolded and built codebase.

```text
User Prompt
    │
    ▼
Architect Agent (Conversational requirements gathering)
    │
    ▼
Requirement Extraction (Extracts features, stack preferences, target users)
    │
    ▼
Architecture Generation (Synthesizes Tech Stack & Major Components)
    │
    ▼
Task Planning (Decomposes architecture into dependency-ordered DAG)
    │
    ▼
Workspace Creation (Allocates filesystem target & initializes metadata)
    │
    ▼
Builder Execution (Generates code files per atomic task specification)
    │
    ▼
Security Validation (Scans file paths and enforces command sandbox)
    │
    ▼
Completion (Marks workspace ready for runtime review)
```

### Stage Breakdown:
1. **User Prompt & Architect Interaction**: The user describes what they want to build via the `/architect` chat interface.
2. **Requirement Extraction**: The `requirement_extractor.py` module parses conversation turns to build a cumulative specification profile (confidence score 0–100%).
3. **Architecture Generation**: Once confidence hits ≥75% (or the user manually forces approval), `spec_engine.py` generates a JSON architecture document.
4. **Task Planning**: `planner_agent.py` and `task_decomposer.py` break the architecture into atomic execution steps.
5. **Workspace Creation**: `workspace_service.py` creates a target folder under `./workspace/<id>` and writes foundational manifest files.
6. **Builder Execution**: `build_orchestrator.py` feeds tasks to `builder_service.py`, generating actual source files.
7. **Security & Completion**: Output is validated against path traversal filters and marked completed.

## 2. Build Pipeline

The Build Pipeline defines how an individual task specification is transformed into functional source code on disk.

```text
Task Specification ──► Build Orchestrator ──► Context Collector (MCP File Read)
                                                        │
                                                        ▼
Generated Files On Disk ◄── File Writer (MCP) ◄── LLM Generation (Builder Agent)
```

### Stage Breakdown:
1. **Task Selection**: The orchestrator selects the next pending task whose dependencies are fully resolved.
2. **Context Collection**: The Builder agent uses `file_mcp.py` to inspect existing workspace files, schemas, and READMEs to maintain codebase consistency.
3. **Prompt Assembly**: The task description, project architecture, and retrieved code snippets are combined into a structured generation prompt.
4. **LLM Generation**: The model (`Gemini 3.1 Pro`) returns JSON-encoded code blocks containing target file paths and complete contents.
5. **Safe File Writing**: `file_mcp.py` validates the target path through `path_security.py` and writes the code to disk.
6. **Telemetry Emission**: WebSocket events notify the frontend to update the live file tree and code diff preview.

## 3. Task Pipeline

Governs the lifecycle of an individual development task.

```text
[Created] ──► [Stored in SQLite/State] ──► [Assigned to Builder] ──► [Executing] ──► [Completed]
                                                                          │
                                                                          ▼
                                                                     [Failed] ──► [Diagnostics]
```

### Stage Breakdown:
1. **Creation**: Generated by `task_decomposer.py` during architecture approval.
2. **Storage**: Persisted via `project_state.py` with unique IDs, titles, descriptions, and dependency links.
3. **Assignment**: Queued by `build_orchestrator.py` when prerequisite tasks reach `Completed` status.
4. **Execution**: Marked `Running`. The Builder agent actively synthesizes the requested feature.
5. **Completion / Failure**: If successful, marked `Completed`. If syntax errors or tool failures occur, marked `Failed` and routed to `FailureDiagnostics`.

---

## 4. Workspace Pipeline

Governs the lifecycle of isolated project environments.

```text
Creation ──► Loading ──► Persistence ──► Real-time Updates ──► Deletion / Archiving
```

### Stage Breakdown:
1. **Creation**: `createWorkspace()` creates directory structure and initializes `workspace.json`.
2. **Loading**: Reads existing workspace manifests from disk and mounts them into runtime memory upon application startup.
3. **Persistence**: Every file edit or status change triggers synchronous writes to database state and filesystem metadata.
4. **Updates**: File modifications trigger broadcast envelopes over the `/ws/workspaces` channel.
5. **Deletion**: Removes local directory tree and archives task records upon user confirmation.

## 5. Conversation Pipeline

Governs real-time conversational interactions with the Architect persona.

```text
Message Input ──► Intent Detection (classify_intent)
                        │
         ┌──────────────┴──────────────┐
         ▼                             ▼
  [Greeting / Help]            [Project Request]
         │                             │
         ▼                             ▼
  Direct Conversational Reply   Requirement Extraction
                                       │
                                       ▼
                                Architect Planning
```

### Stage Breakdown:
1. **Message Input**: User submits text via `Architect.jsx`.
2. **Intent Detection**: `architect_service.py` evaluates keywords before initiating expensive LLM calls.
   - **Conversational Intents** (`hi`, `help`): Returned immediately with helpful guidance.
   - **Project Intents**: Routed to deeper analysis pipeline.
3. **Requirement Extraction**: Parses input to update project requirements and increment confidence score.
4. **Architect Planning**: Formulates follow-up clarifying questions or recommends project approval.
